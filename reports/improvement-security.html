Vulnerability

SEC-06HIGHAuthZ

JWT filter reads role from token claim, not DB — role change takes up to 15 min to propagate

Vulnerability

The filter loads the User entity from DB (for isActive) but then uses the role JWT claim to build the security context. After an admin changes a user's role, the old role remains effective for up to 15 min (access token TTL). A downgraded ADMIN keeps ADMIN privilege; a promoted RESIDENT keeps RESIDENT privilege until their next login.

Recommendation

One-line fix: replace roleFromToken with user.getRole().name() on line 134. The DB entity is already loaded on line 119. This makes role changes effective on the next request after the admin update, with zero TTL lag.

SEC-09MEDIUMConfig

Vulnerability

SEC-10MEDIUMSecrets

Admin BCrypt hash committed to version control — offline dictionary attack surface

Vulnerability

The admin account's BCrypt hash is committed to git in V2__seed_admin.sql. While cost 12 is strong, the committed hash enables targeted offline attack. Anyone with repo access can begin a dictionary attack without DB access. The migration comment instructs a password change on first login but does not enforce it.

Recommendation

Use Flyway placeholder support: replace the hash with ${ADMIN_PASSWORD_HASH} in the SQL, injected at migration time from an environment variable. Alternatively, seed with a forced-reset-on-login flag so the default is known-weak but cannot be retained after first login.

SEC-13MEDIUMAuthZ

Logout without Authorization header silently skips token blocklisting

Vulnerability

When the Authorization header is absent, extractBearerToken returns "". The service calls extractJti(""), which throws JwtException caught at DEBUG level — the blocklist write is skipped silently. A client that calls logout without the header while retaining a valid token has an unblocklisted token indefinitely.

Recommendation

Return null (not "") from extractBearerToken when the header is absent. Guard the blocklist path in logout(): if (accessToken == null || accessToken.isBlank()) return; before JTI extraction. Or respond 400 if the header is missing.

SEC-14MEDIUMConfig

Redis KEYS command in logout — O(N) blocking scan; blocks entire Redis server

Vulnerability

redisTemplate.keys(pattern) maps to the Redis KEYS command, which is O(N) over all keys and blocks the entire server for its duration. In production with many JWT and rate-limit keys, every logout stalls Redis — including concurrent token validation calls for all active users.

Recommendation

Replace with cursor-based scan: redisTemplate.scan(ScanOptions.scanOptions().match(pattern).count(100).build()). Collect cursor results into a list, then call redisTemplate.delete(keys). This is the standard non-blocking replacement for KEYS in production.

Evidence
String pattern = REFRESH_KEY_PREFIX + principal.getId() + ":*";
var keys = redisTemplate.keys(pattern);  // ← Redis KEYS — O(N) blocking
if (keys != null && !keys.isEmpty()) {
    redisTemplate.delete(keys);
}
