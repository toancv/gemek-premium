B-01 — Unsanitised groupBy parameter — logic / query injection

✓ FIXED
Finding

B-02 — Unsanitised sort field in UserController — ORDER BY injection

✓ FIXED
Finding

The sort query parameter was used to construct a JPA Sort object directly from raw user input. A malicious value could inject SQL into the ORDER BY clause of the generated query.

HIGH

B-04 — X-Forwarded-For blindly trusted — rate-limit bypass via IP spoofing

✓ FIXED
Finding

The login rate limiter used X-Forwarded-For without verifying the immediate connection source. An external attacker could set X-Forwarded-For: 127.0.0.1 to bypass per-IP limits entirely. A follow-up finding noted that the initial RFC 1918 check incorrectly matched 172.0.0.0/8 instead of the correct 172.16.0.0/12.

Fix Applied (2 cycles)

6e43ebf: Header only trusted when RemoteAddr is a private/loopback address.
6519fa7: Corrected isPrivateOrLoopback() to cover RFC 1918 ranges exactly: 10.x, 172.16–31.x, 192.168.x, 127.x, ::1.

AuthServiceImpl.java (getClientIp · isPrivateOrLoopback) commits 6e43ebf + 6519fa7 Re-scan: RESOLVED

B-09 — Missing HTTP security headers (XSS / clickjacking / MIME sniff / HSTS)

✓ FIXED
Finding

No security-relevant HTTP response headers were set. Missing Content-Security-Policy, X-Frame-Options, X-Content-Type-Options, and Strict-Transport-Security leaves the API responses vulnerable to XSS escalation, clickjacking, MIME confusion, and protocol downgrade.

Fix Applied

Added via Spring Security http.headers() DSL in SecurityConfig:
• X-Content-Type-Options: nosniff
• X-Frame-Options: DENY
• Strict-Transport-Security: max-age=31536000; includeSubDomains
• Content-Security-Policy: default-src 'self'

✓ FIXED

Fix Applied

HIGH

F-02 — Auth route guard not reactive — expired token does not trigger logout

✓ FIXED
Finding

The Vue Router navigation guard checked token presence only at route navigation time. If a token expired mid-session during a long idle period, the UI remained on protected pages without redirecting to login, and subsequent API calls used the expired token.

Fix Applied

Replaced the static check with a reactive guard that verifies token expiry on every navigation and on a periodic interval. Expired state triggers immediate redirect to /login and clears the Pinia token store.

F-04 ≡ SEC-20 — Refresh token stored in localStorage — readable by XSS

✓ RESOLVED
Finding

The refresh token is persisted in localStorage. Any XSS vulnerability in the SPA, including from third-party scripts, can read localStorage and exfiltrate the refresh token, enabling long-lived account takeover even after session close. Same finding as SEC-20 (logged INFO); double-counted — unified here, MEDIUM governs.

Remediation (Hardening Sprint — DONE)

H3 (BE): /auth/login and /auth/refresh set Set-Cookie: refreshToken=...; HttpOnly; SameSite=Strict; Path=/api/auth (Secure profile-driven via app.auth.cookie-secure, true in prod). H4 (FE): both apps removed every localStorage read/write of the refresh token; refresh is cookie-implicit (empty body + X-Requested-With). Close-out: the dual-mode body channel removed — refresh token is cookie-only, no longer present in any JSON response body (test-enforced).
